# 股票实时行情抓取与短线异动推荐 —— 设计文档

日期:2026-07-16
状态:待实现

## 1. 背景与目标

现有股票模块(`src/stock/`)的数据全部是日线(腾讯 `fqkline` 为主,东财 `klt=101` 兜底),`StockBar` 只有 `date` 字段、没有时间戳,不存在任何盘中数据通道。

本项目新增:

1. **实时行情抓取** —— 开市日 10:00–11:30、13:30–15:00,每 10 分钟抓取一次全 A 股快照
2. **数据分层保留** —— 原始快照只保留 10 个自然日滚动淘汰;信号与其结局永久保留(5.3)
3. **短线异动推荐** —— 基于盘中突变筛出候选股,并标注「短线」或「长线」
4. **主力资金流检测** —— 识别主力资金大量流入/流出,作为异动佐证与排序依据

### 非目标(明确排除)

- 不做分时明细/逐笔成交
- 不做长线选股(已有 `src/stock/recommend.rs`,不重复造)
- 不做策略验证/回测 —— 本次只产出信号与评估数据,验证是后续独立项目
- 不改动现有日线抓取链路

## 3. 架构

新建 `src/stock/realtime/`,与 `src/stock/data/`(日线)并列,互不干扰。

```
src/stock/realtime/
├── mod.rs         公开接口
├── snapshot.rs    腾讯全市场价量快照(7 请求/次)
├── flow.rs        东财候选股资金流(1 请求/次)
├── calendar.rs    交易日判断
├── store.rs       SQLite 读写 + 10 天清理
└── movers.rs      异动检测 + 资金流佐证 + 短/长线标签
```

数据流:

```
push/schedule.rs (60s tick)
  └─> 到达抓取时点?
        └─> calendar::should_fetch()  ──非交易日──> 跳过
              └─> snapshot::fetch()   腾讯全市场价量(7 请求)
                    ├─> calendar 时间戳自证 ──非今天──> 标记节假日 + 丢弃
                    └─> store::insert_ticks()
                          └─> movers::detect()  异动候选(价量双阈值触发)
                                ├─> flow::fetch(候选)   东财资金流(1 请求)
                                │     └─> movers::annotate()  排序权重 + 背离标记
                                ├─> movers::classify()  短/长线标签
                                ├─> store::insert_signal()  永久留档
                                ├─> push (限流)
                                └─> Web API 随时查
              └─> store::prune()  删除 10 天前的 ticks(不动 signals)

收盘后 / 次日日线同步
  └─> store::backfill()  由日线缓存回填 signals 的 close_ret / ret_t1 / ret_t5
```

## 4. 抓取层

### 4.1 数据源:双源分工

价量与资金流走**两个不同的源**。这不是设计偏好,是 4.2 实测封禁逼出来的唯一可行解。

#### 全市场价量 —— 腾讯 `qt.gtimg.cn`

```
https://qt.gtimg.cn/q=sh600519,sz000001,...
```

- **实测单请求可带 800 个代码**,返回 `v_sh600519="..."` 的 `~` 分隔文本(非 JSON)
- 全市场 5400 只 → **7 请求/快照,140 请求/交易日**,实测无封禁
- 现有 `tencent.rs` 已用腾讯拉日线,代码映射(`sh`/`sz` 前缀)可直接复用

返回字段(按 `~` 分割的下标,实测自 600519):

| 下标 | 含义 | 实测值 |
|---|---|---|
| 1 / 2 | 名称 / 代码 | 贵州茅台 / 600519 |
| 3 | 现价 | 1258.99 |
| 4 | 昨收 | 1251.06 |
| 6 | 成交量(手) | 47611 |
| 30 | **行情时间戳** | `20260716161440` |
| 32 | 涨跌幅(%) | 0.63 |
| 37 | 成交额(万元) | 598757 |
| 38 | 换手率(%) | 0.38 |
| 49 | 量比 | 0.98 |

下标 30 的时间戳即 4.4 快照自证的依据。

⚠ 腾讯**不提供资金流**,这是它唯一的缺口。

#### 候选股资金流 —— 东财 `ulist.np`

```
https://push2.eastmoney.com/api/qt/ulist.np/get?fltt=2&secids=1.600519,0.000001,...&fields=f12,f14,f2,f3,f62,f184
```

- 接受**显式 secid 列表**,只查 6.1 筛出的几十只候选
- **1 请求/快照,20 请求/交易日** —— 远低于封禁线
- 字段:`f62`(主力净流入额)、`f184`(主力净流入占成交额比)

⚠ **量纲陷阱(硬性)**:不加 `fltt=2` 时返回**放大 100 倍的定点整数**。实测 600519 `f2=125899`(实为 1258.99)、`f184=-132`(实为 -1.32%)。必须显式带 `fltt=2`,并在解析层加量纲哨兵测试。此坑 `universe.rs:236-239` 已有明文警告。

资金流盘中返回**当日累计值**。

#### 为什么资金流只取候选股

原设计对全市场 5400 只取资金流,但按 6.3,资金流仅用于候选股的佐证/排序/背离标记 —— **其中 99% 的数据永不会被读取**。只取候选既规避封禁,又消除了这份浪费。

连带影响:`ticks` 表**不含资金流字段**(见 5.2),资金流仅存于 `signals`。

#### 复用现有基建

- 腾讯路径:复用 `tencent.rs` 的代码前缀映射
- 东财路径:复用 `universe.rs:119` 的 `client()` 与 `:129` 的 `get()`(`pub(crate)`,`valuation.rs:22` 已跨模块复用)。**不得再手写 reqwest builder** —— 二者已封装 IPv4 强制绑定(东财部分 IPv6 CDN 节点不可达)、20s 超时、4 次指数退避重试

### 4.2 数据源封禁:实测记录

> 本节记录 2026-07-16 的实测结果。它推翻了本设计初稿「东财 clist 一次请求拉全市场」的方案,是 4.1 双源分工的**唯一动因**。后续维护者若想「简化」回单源东财,请先重读本节。

| 端点 | 实测结果 | 资金流 |
|---|---|---|
| 东财 `clist`(批量清单) | **封禁 `http=000`** | ✓ |
| 东财 `ulist.np`(批量指定) | 初期 200 ✓,封禁蔓延后 `http=000` | ✓ |
| 东财 `datacenter`(估值表) | 200 ✓,不受影响 | ✗(且无实时) |
| 东财 `push2/qt/stock/get`(单只) | 200 ✓ | ✓ |
| **腾讯 `qt.gtimg.cn`** | **200 ✓,800 只/请求** | ✗ |

实测细节:

- **不到 15 次请求即触发封禁**。首次 `pz=200` 成功,第二次 `pz=6000` 被拒,此后连原本成功的 `pz=200` 也全部 `http=000`
- **封禁会蔓延**:先 `clist`,继而 `ulist.np` 亦不可用
- **封禁持续 ≥600 秒**:以 30 秒间隔连测 20 次,全程 `http=000`,**观测窗口结束时仍未解封**。600 秒 = 本项目的整个抓取间隔,意味着一旦触发,后续多个时点会连续失效
- ⚠ **封禁可能被请求本身续期**:上述监测在封禁期间持续发请求。故「600 秒未解」无法区分「固定时长 ≥600s」与「每次请求都在续封」。**实现按后者假设处理**(见下)
- 封禁期间 `datacenter` 与单只 `stock/get` 正常 —— 故封禁针对批量行情端点,非全站

此结果与 `universe.rs:6-20` 记录的历史结论一致:作者当初正因 clist「翻到第 5 页即被掐、封禁持续」才将 A 股清单迁至 datacenter。

**硬性约束**:5400 只按 clist `pz=200` 需翻 27 页 × 20 时点 = **540 页请求/日**,必封。clist 路径在本项目中**禁止使用**。

#### 资金流请求禁止重试(硬性)

`universe::get` 会重试 4 次带指数退避。那对全市场清单是对的(残缺清单是静默错误,宁可重试也要拿到),但对资金流是**有害的**:

1. 资金流失败**本来就允许降级** —— 佐证拿不到不影响价量主判定成立
2. 封禁期间重试 = 一次调用打 4 发,只会加深伤口

故 `flow.rs` **自建 client、单发不重试**、超时压到 8s(封禁时连接是直接被掐的,等 20s 无意义)。**对一个可失败的功能,快速失败比顽强重试正确。**

### 4.3 抓取时点

| 窗口 | 时点数 |
|---|---|
| 10:00 – 11:30 | 10(10:00, 10:10, … 11:30) |
| 13:30 – 15:00 | 10(13:30, 13:40, … 15:00) |
| 合计 | **20 / 交易日** |

接入 `push/schedule.rs` 现有 60 秒阻塞 tick 循环。

### 4.4 交易日判断(`calendar.rs`)

两道闸,零维护:

1. **周末本地跳** —— `chrono` 判定周六/周日,不发任何请求
2. **快照自证** —— 工作日发一次快照,比对返回的行情时间戳日期是否为今天。不是 → 判定为节假日,丢弃本次数据,将当日写入 `non_trading_days` 表,当天后续 19 个时点不再重试

⚠ **「数据陈旧」≠「今天是节假日」**(实现期踩过,是个静默失败):

- **开盘后**数据仍陈旧 → 今天确实是节假日,可标记
- **开盘前**数据陈旧 → 完全正常,市场还没开而已

实测:00:03(周五,真实交易日)跑一次抓取,数据源理所当然返回昨天的收盘数据,自证正确判定「非今日」,代码却据此把当天永久标记成节假日 —— 开市后守护查到「今天是节假日」,**整个交易日被静默跳过**。

故标记前须再过一道闸:仅当 `now ≥ 09:30`(开盘)时,陈旧才意味着节假日。此闸放在 `run_tick` 内而非调用方 —— 任何调用方都不该有能力毒化整个交易日。

**为什么不用硬编码节假日表**:每年年底必须手工更新,忘记则静默抓错。

**为什么不用 `universe::latest_trade_date()`**:它返回「最近已收盘日」,盘中取到的是昨天,无法回答「今天开不开市」。

## 5. 存储层

### 5.1 库文件

新建 `data/realtime.db`,与 `data/xlh.db`(账号/授权)物理隔离。

理由:
- 盘中每 10 分钟写 5400 行,不应与登录会话争同一 WAL 锁
- 纯派生数据,损坏可直接删除重建,不涉及用户资产

路径从 config 读取(**不得**照抄 `web/stock.rs` 的 `&'static Path` 硬编码)。

### 5.2 表结构

#### ⚠ `ts` 列的口径

存的是 `naive_local.and_utc().timestamp()` —— 把**本地**朴素时间当 UTC 编码,读回时反向解,精确互逆。全模块统一此口径(写入、读取、prune 的时间比较、`pushed_today` 的日界),内部完全自洽;A 股只有一个时区,不需要真正的时区换算。

但**它不是真 Unix 时间戳**:外部脚本用 `datetime.combine(day, t).timestamp()`(本地→epoch)写入本表会差 8 小时(CST=UTC+8)。实现期踩过 —— 注入的 10:30 显示成了 02:30。

**`ticks`** —— 全市场价量快照(**来源:腾讯,不含资金流**)

| 列 | 说明 |
|---|---|
| `code` | 股票代码,联合主键 |
| `ts` | Unix 秒,联合主键 |
| `price` | 现价 |
| `change_pct` | 当日涨跌幅 |
| `volume` | 成交量(手) |
| `amount` | 成交额(元) |
| `turnover` | 换手率 |
| `vol_ratio` | 量比 |

`(code, ts)` 联合主键天然去重:同一时点重试不会写重。

**此表刻意不含资金流字段**:资金流仅对候选股取(4.1),全市场 99% 的股票没有该数据。资金流存于 `signals`。

**`signals`** —— 触发过的异动信号及其结局(**永久保留**)

| 列 | 说明 |
|---|---|
| `id` | 主键 |
| `code` / `name` | 股票 |
| `ts` | 触发时刻(Unix 秒) |
| `trigger_price` | 触发时价格 |
| `jump_pct` | 触发时 10 分钟突变幅度 |
| `vol_surge_x` | 触发时量能放大倍数 |
| `main_net` | 触发时主力净流入额(东财 `f62`,当日累计) |
| `main_net_pct` | 触发时主力净流入占成交额比(东财 `f184`,当日累计) |
| `divergence` | 背离标记(见 6.3) |
| `horizon_tag` | 短线 / 长线 |
| `pushed` | 是否推送过(用于限流与事后分析) |
| `close_ret` | 结局:当日收盘价相对 `trigger_price` 的涨跌 |
| `ret_t1` / `ret_t5` | 结局:T+1 / T+5 收盘相对 `trigger_price` 的涨跌 |

**`non_trading_days`** —— 快照自证判定出的节假日,一行一个日期。

### 5.3 分层保留策略

原始快照与信号记录是两种性质完全不同的数据,保留期必须分开:

| 表 | 体量 | 用途 | 保留期 |
|---|---|---|---|
| `ticks` | 108 万行滚动 | 算异动、算量能基准 | **10 自然日** |
| `signals` | ~几十行/天,≈1 万行/年 | 验证阈值有效性 | **永久** |

#### `ticks` —— 10 自然日

每个抓取周期结束后:

```sql
DELETE FROM ticks WHERE ts < ?;   -- 阈值 = 当前时刻 − retain_days 自然日
```

删除后执行 `PRAGMA incremental_vacuum`。**不可省略** —— 否则 SQLite 仅将页标记为空闲、文件不缩,长期运行磁盘只涨不落。

稳态数据量:5400 只 × 20 时点 × 10 天 ≈ **108 万行 / 80–120 MB**(8 列,无资金流)。

**10 天口径 = 自然日**(非交易日)。理由:遇长假(如春节)时,自然日窗口仍能保留节前数日数据;交易日口径会让窗口跨度失真。

⚠ **自然日口径下,10 天窗口内的实际交易日数会波动**:平常约 6–8 个,长假期间可能低至 3–4 个。量能基准的样本数因此不是恒定的,`baseline_days` 只是上界(见 6.1)。

**为什么不再延长到 30/90 天**:延长仅为让 6.1 的量能基准多几个同时段样本,收益递减(中位数在 ~10 个样本后趋稳);代价是库涨到数百万行、数百 MB,而其中 99.9% 是「未触发任何信号的普通股票的普通快照」,写入后永不再读。验证阈值靠 `signals` 表,不靠堆原始数据。

#### `signals` —— 永久

一年约 1 万行、数 MB,不构成任何存储压力。

**永久保留是硬性要求**:第 10 节列为头号风险的「阈值是猜的,信号可能全是噪音」,唯一的解法是积累足量样本后做统计检验。信号样本需数百至上千个,按每天几十个计需数月。若 `signals` 随 `ticks` 一起滚动删除,则**永久失去回答「这套阈值是否有效」的能力**。

**结局字段不依赖 `ticks`**:`close_ret` / `ret_t1` / `ret_t5` 均由 `trigger_price` 与**日线缓存**(`.cache/stock/*.csv`,永久保留)比对算出。因此 raw 数据 10 天后删除,不影响日后回溯任意时间跨度的结局,亦可随时补算新的结局口径(如 T+20)。

## 6. 异动检测、资金流与短/长线标签

### 6.1 异动判定(`movers.rs`)

对每只股票取库中最近两个快照,计算:

- **10 分钟价格突变** = `(现价 − 上一时点价) / 上一时点价`
- **量能放大倍数** = `本时点成交量增量 ÷ 过去 baseline_days 天内同一时点成交量增量的中位数`

⚠ **基准必须是「增量」而非「累计量」**(实现期踩过):数据源给的 volume 是**当日累计**。基准若取历史同时点的累计量,就成了「拿增量除以累计量」—— 不仅无意义,还随时间越来越离谱(尾盘累计量是早盘的十几倍,同一增量在 14:50 算出的倍数比 10:10 小一个量级)。实测对照:增量 600 配基准 500 得 1.2 倍(正确),配累计量 20500 得 0.03 倍 —— 量能判据被彻底稀释,任何真实异动都无法触发。

派生的三条实现约束:

- 每个历史日**独立做差**(累计量每天从 0 重来,跨日做差得巨大负数)
- 当日**首个时点跳过**(无前值;若误当增量,它等于开盘至今全部成交量,是个巨大的假基准)
- **负增量丢弃**(数据源偶发累计量回退)

**两者同时超阈值**才判定为异动。

理由:只看价不看量 → 捞进开盘半小时的正常波动;只看量不看价 → 捞到大宗横盘。

#### 量能基准为何用「同时段历史」而非「当日均值」

A 股成交量有显著的 U 型日内分布:开盘半小时与尾盘放量,午间萎缩。若以当日累计均值为基准,则 10:00 的基准被开盘高量拉高(异动难触发)、14:00 的基准被午间低量拉低(异动易误触发)—— **同一套阈值在一天之内的实际严格程度会漂移**。

以「历史同一时点的中位数」为基准,则 10:00 与历史 10:00 比、14:00 与历史 14:00 比,U 型分布被自然消去。用中位数而非均值,避免单日异常值污染基准。

这正是 `ticks` 保留原始快照的**用途所在**。

#### `baseline_days` 与 `retain_days` 解耦

二者是**独立配置项**,不是同一个数:

- `retain_days`(10)—— 存多久,受磁盘约束
- `baseline_days`(10)—— 基准回看多远,受**统计与时效的权衡**约束

**约束**:`baseline_days ≤ retain_days`。启动时校验,违反则报错退出 —— 否则基准会静默地只用到实际存在的数据,行为与配置不符。

**为什么解耦**:窗口拉长 → 中位数更稳、抗噪更好;但也更迟钝 —— 一只刚进入活跃期的股票,其「正常量」已抬升,过长的窗口仍拿冷清期的旧量当基准,会把它的每一个时点都误判成异动。两者是真实的取舍,不该被「存了多少天」这个磁盘决策绑架。

默认设为 10(与 `retain_days` 相等,用满数据),但**保留单独调小的能力** —— 若上线后发现活跃股持续误报,把 `baseline_days` 调到 5 是第一个该试的动作,不必动 `retain_days`。

**冷启动降级**:首次部署或次新股上市不足 `baseline_days` 时,历史样本不足。此时降级为当日累计均值基准,并在该信号上标记 `baseline=fallback`。样本攒够后自动切换。不得因样本不足而静默跳过检测。

⚠ 因 `retain_days` 是自然日口径,窗口内实际交易日数会波动(平常 6–8 个,长假期间可能仅 3–4 个)。故 `baseline_days` 是**上界而非定数**,实现须容忍样本数不足 `baseline_days`,仅当样本数为 0 时才走 fallback。

### 6.2 阈值配置化(硬性要求)

阈值**不得写死在代码中**,置于 `config.toml` 新增 `[realtime]` 段。

理由:`stock_advice.rs:3-11` 中作者自述其 10%/30% 系「拍脑袋的魔数」,且基金侧同款择时逻辑上线后被证伪删除。可配置化意味着可在积累数周真实数据后用 `evidence.rs` 前瞻检验框架调参,而非改代码重编译。

初始值:价格突变 ±2%,量能放大 3 倍。**此为待验证起点,不是结论。**

### 6.3 主力资金流(第三佐证)

资金流**不是触发条件**,触发仍完全由 6.1 的价量双阈值决定。

理由有二:

1. 东财资金流数据需先按单笔金额分档统计后推送,比价量慢半拍。设为必要条件会漏掉「已拉升但资金流数据尚未跟上」的信号。
2. **架构上不可能** —— 资金流只对候选股取(4.1),而候选正是由价量双阈值筛出的。触发发生在拿到资金流之前。

**获取路径**:`movers::detect()` 筛出候选后,`flow::fetch(&candidates)` 用东财 `ulist.np` 一次请求取回这几十只的 `f62`/`f184`(4.1)。

**降级(硬性)**:东财封禁期间(4.2 实测 ≥180 秒,可跨多个时点)`flow::fetch` 会失败。此时**必须照常出榜**,资金流字段留空,`signals.main_net*` 写 NULL,不得因资金流缺失而丢弃已确认的价量异动 —— 资金流是佐证,佐证拿不到不影响主判定成立。榜单/推送对该批标注「资金流暂不可用」。

资金流的三个用途:

1. **排序权重** —— 同为异动,主力净流入占比高的排前面
2. **展示** —— 榜单/推送中标注资金流方向与强度
3. **背离标记** —— 资金流方向与价格方向相反时显式标出:
   - 涨 + 主力净流出 → 疑似散户抬轿
   - 跌 + 主力净流入 → 疑似主力吸筹

   这是最有信息量的情况,单看价量看不出来。

**「大量」的口径 = 主力净流入 / 当日成交额**(东财 `f184` 直接给出),阈值置于 `[realtime]` 配置段,初始 ±5%,**待验证**。

**为什么用相对口径而非绝对额**:5400 只股票市值差数百倍。1 亿净流入对贵州茅台是噪音,对小盘股是巨量。绝对额口径会让榜单被大市值股长期占据。

#### 局限(硬性:必须体现在所有出口的声明中)

东财的「主力资金」**不是真实席位数据**,是按单笔成交金额分档**推算**的代理指标:

- 超大单(>100 万)、大单(20–100 万)→ 归为「主力」
- 中单、小单 → 归为「散户」

它**无法**区分机构、游资、拆单大户,**无法**识别对倒。与 6.4 的短/长线标签同属「线索」,不是「事实」。

### 6.4 短/长线标签

异动仅负责筛候选;标签由现有模块判定:

- 拉取该股 `valuation.rs` 的 PE/PB 历史分位
- 拉取 `diagnose.rs` 的 MA20/MA60 趋势

判定规则:

| 标签 | 条件 |
|---|---|
| **长线** | 估值处于历史低分位 **且** 趋势非下跌 → 异动有基本面支撑 |
| **短线** | 其余全部 → 纯消息面/情绪驱动 |

候选每时点仅数十只,逐只拉取完全可承受,无需对 5400 只全算。

### 6.5 局限声明

短/长线标签为启发式,与 `stock_advice.rs` 的买卖阈值同样**未经前瞻检验**。它是「分类线索」,不是「已验证策略」。真正验证需在第 5 节数据积累数周后另开一轮回测 —— 属后续项目,不在本次范围。

资金流的独立局限见 6.3 末节。

## 7. 交付层

三个出口共用同一份库数据。

### 7.1 Web API

新增 `GET /api/stock/realtime/movers`,返回当前异动榜:候选股 + 短/长线标签 + 触发时价量数据 + 主力资金流方向与强度 + 背离标记。

风格对齐现有 `src/web/stock.rs`:JSON API,`spawn_blocking` 包裹(数据层为 `reqwest::blocking`)。

### 7.2 盘中推送(限流)

接入现有飞书通道(`push/channels.rs`)。三层闸:

1. 同一只股票**当日只推一次** —— 已推记于内存 set,收盘清空
2. 只推**强信号**(≥ 阈值 1.5 倍),弱信号仅进库
3. 每时点**最多 5 只** —— 超出时按 6.3 的资金流权重排序取前 5

无此三层则一天 20 时点 × 数十只 = 数百条,飞书必被刷屏至静音。

### 7.3 收盘汇总

15:00 后**先回填 `signals` 表的 `close_ret` 字段**(由日线缓存的当日收盘价与 `trigger_price` 比对得出),再推送一条当日汇总,含:

- 当日触发过的全部异动股
- 各自标签(短/长线、资金流方向、背离标记)
- 触发时突变幅度
- 当日主力净流入占比(收盘值)
- **收盘价相对触发点的涨跌**

最后一项是重点:它免费积累出信号质量评估数据 —— 异动信号发出后至收盘是涨是跌,数月后即可回答「这套阈值是否有效」。

**汇总消息只是数据的展示,不是数据本身** —— 结局必须落 `signals` 表(5.2),推完即焚等于没做。

`ret_t1` / `ret_t5` 由后续交易日的日线同步流程顺带回填,不需要独立调度。

### 7.4 免责声明

每个出口均带免责声明,与现有 `StockRecommendReport.disclaimer` 一致。声明须包含 6.3 的资金流代理指标局限 —— 不可让读者误以为「主力资金」是真实席位数据。

## 8. 配置

`config.toml` 新增:

```toml
[realtime]
db_path = "data/realtime.db"
retain_days = 10           # 仅作用于 ticks;signals 永久保留,无对应开关
baseline_days = 10         # 量能基准回看窗口;须 ≤ retain_days,启动校验
price_jump_pct = 0.02      # 价格突变阈值,待验证
volume_surge_x = 3.0       # 量能放大阈值(相对同时点历史中位数),待验证
main_flow_pct = 0.05       # 主力净流入占成交额比,「大量」阈值,待验证
strong_signal_x = 1.5      # 强信号 = 阈值的几倍
max_push_per_tick = 5
```

`signals` 的保留期**刻意不做成配置项** —— 它是验证阈值的唯一依据(5.3),不应存在「一改配置就把它清掉」的路径。

## 8.1 CLI

守护会自动跑,但 `realtime` 子命令让这条链路随时可被**真实驱动**:

| 命令 | 用途 |
|---|---|
| `xlh realtime once [--limit N]` | 立即抓一次并检测,打印榜单。`--limit` 用于快速验证 |
| `xlh realtime movers [--day D]` | 打印当日异动榜 |
| `xlh realtime summary [--day D]` | 回填结局并打印收盘汇总 |

**存在的理由**:本项目最严重的两个 bug(盘前毒化交易日、基准量纲错配)单测全绿、只有真跑才暴露。没有这组命令,它们会一路活到生产。

## 9. 测试

| 层 | 方式 |
|---|---|
| `calendar.rs` | 纯函数:给定日期/时间戳 → 是否抓取。周末、节假日、正常交易日、时间戳非今天 |
| `store.rs` | 内存 SQLite:写入去重、`retain_days` 边界清理(第 10 天保留、第 11 天删除)、**清理 `ticks` 不得波及 `signals`** |
| `movers.rs` | 构造 tick 序列:仅价格超阈值不触发、仅量能超阈值不触发、两者超才触发;资金流不影响是否触发、只影响排序;背离标记(涨+净流出、跌+净流入)正确置位 |
| 量能基准 | 同时点历史中位数计算正确;单日异常值不污染中位数;样本数少于 `baseline_days`(长假)仍正常计算;样本数为 0 时降级为当日均值并标记 `baseline=fallback`,不得静默跳过 |
| 配置校验 | `baseline_days > retain_days` 时启动报错退出,不得静默降级 |
| 结局回填 | 给定 `trigger_price` + 日线,`close_ret`/`ret_t1`/`ret_t5` 计算正确;日线缺失时留空而非写 0 |
| `snapshot.rs` | 用固化的**腾讯** `~` 分隔响应样本测解析,不打网络;下标 30 时间戳、下标 3/6/32/37/38/49 各字段取值正确;停牌股返回空串时不 panic |
| `flow.rs` | 用固化的东财 `ulist.np` 响应测解析;**量纲哨兵**:`fltt=2` 下 `f2` 须为浮点真值(1258.99),若样本呈 125899 则测试失败 —— 防东财改量纲后静默算错 |
| 封禁降级 | `flow::fetch` 返回 Err 时,榜单照常产出、`main_net*` 为 NULL、标注「资金流暂不可用」,不得丢弃候选 |
| 推送限流 | 同股当日二次触发不推、弱信号不推、超 5 只截断 |

## 10. 风险

| 风险 | 缓解 |
|---|---|
| 阈值是猜的,信号可能全是噪音 | 阈值配置化;`signals` 表永久保留信号+结局(5.3),数月后可做统计检验;文档明示未验证 |
| `signals` 被误当缓存清理掉 | 一旦删除即永久失去验证能力。5.3 明示永久保留;测试覆盖「清理 `ticks` 不波及 `signals`」 |
| **「主力资金」被误读为真实席位数据** | 它是按单笔金额分档推算的代理指标(6.3),识别不了机构/游资/对倒。仅作佐证不作触发;所有出口的声明须写明 |
| **东财封禁**(已实测发生,非假设) | 全市场价量改走腾讯,东财日请求量从 540 降至 20(4.1);封禁时资金流降级留空、榜单照出(6.3);抓取失败仅跳过该时点,不影响日线链路 |
| 腾讯接口变更/限流 | 现有 `tencent.rs` 已依赖腾讯,风险已存在于日线链路;抓取失败跳过该时点。⚠ 腾讯成为**全市场价量的单点依赖**,无兜底源(东财批量不可用) |
| 东财 `ulist.np` 量纲变更 | 解析层量纲哨兵测试(第 9 节);`fltt=2` 必须显式带 |
| 库文件膨胀 | `incremental_vacuum` 强制回收 |
| 刷屏 | 三层限流闸 |
